# 완성 정의 및 수용 기준

이 문서는 Codex가 구현을 완료했다고 판단할 수 있는 명확한 기준이다. 구현은 단순 MVP가 아니라, 아래 수용 기준을 통과하는 **행사 당일 운영 가능한 완성본**을 목표로 한다.

## 1. 완성 정의 Definition of Done

프로젝트는 다음 조건을 모두 만족해야 완성으로 본다.

1. Next.js 15, TypeScript, Tailwind CSS 기반 앱이 정상 실행된다.
2. Supabase SQL 스키마와 seed 데이터가 제공된다.
3. 30개 팀 seed 데이터가 생성된다.
4. 16개 공식 미션 seed 데이터가 생성된다.
5. 히든 QR 10개 seed 데이터가 생성된다.
6. 팀 코드 기반 로그인으로 참가자 화면에 접근할 수 있다.
7. 팀 대시보드에서 점수, 추첨권, 클리어 테마, 생명의 열쇠 코드 조각이 보인다.
8. 미션 코드 입력으로 16개 미션 상세 화면에 접근할 수 있다.
9. quiz, text, photo, screenshot, staff, final, easter_qr 타입 미션이 각각 동작한다.
10. 같은 팀이 같은 미션을 여러 번 제출해도 점수가 중복 지급되지 않는다.
11. 한 테마에서 1개 이상의 미션 성공 시 해당 테마가 클리어된다.
12. 8개 테마 클리어 시 완주 보너스 100점이 지급된다.
13. 16개 전체 미션 완료 시 올클리어 보너스 200점이 지급된다.
14. 8개 테마 클리어 시 “홍명기홀로 오라!” 문구가 표시된다.
15. 8개 테마를 클리어한 팀만 홍명기홀 최종 인증에 성공한다.
16. 최종 인증 성공 시 추첨권 2장이 추가된다.
17. 추첨권은 점수 구간과 최종 인증 여부에 따라 최대 6장으로 계산된다.
18. 히든 QR은 팀당 최대 3개까지만 점수 인정된다.
19. 숨은 운영진 보너스는 팀당 1회만 인정된다.
20. 관리자 화면에서 팀 현황, 제출 내역, 승인 대기, 스태프 미션 승인, 점수 수정, 공지 발송이 가능하다.
21. 관리자 액션은 audit log에 기록된다.
22. 사진 업로드는 클라이언트 압축 또는 업로드 제한 정책을 포함한다.
23. CSV export 또는 운영 백업용 데이터 내보내기 기능이 있다.
24. Playwright E2E 테스트가 주요 사용자 흐름을 검증한다.
25. k6 부하 테스트 스크립트가 제공된다.
26. README에 로컬 실행, Supabase 설정, seed 실행, 테스트 실행, 배포 방법이 문서화되어 있다.

## 2. 핵심 사용자 흐름 수용 기준

## 2.1 팀 로그인

### Given

팀 seed 데이터에 `TEAM-01-KEY` 로그인 코드가 있다.

### When

사용자가 `/login`에서 팀 코드를 입력한다.

### Then

- 로그인에 성공한다.
- `/dashboard`로 이동한다.
- 팀명과 현재 점수 0점이 보인다.
- 클리어 테마 0/8이 보인다.
- 생명의 열쇠 코드는 `_ _ _ _ _ _ _ _` 또는 유사한 미완성 상태로 보인다.

## 2.2 미션 코드 입력

### Given

팀이 로그인되어 있다.

### When

`/code`에서 `NUT-30`을 입력한다.

### Then

- `/mission/NUT-30` 또는 해당 미션 상세 화면으로 이동한다.
- 미션명, 점수, 설명, 제출 방식이 보인다.

## 2.3 퀴즈 미션 성공

### Given

팀이 `NUT-30` 미션에 접근했다.

### When

퀴즈 정답을 통과 기준 이상 제출한다.

### Then

- 제출이 approved 처리된다.
- 30점이 반영된다.
- Nutrition 테마가 클리어된다.
- 생명의 열쇠 코드 조각 `홍`이 공개된다.
- 같은 미션을 다시 제출해도 추가 점수는 지급되지 않는다.

## 2.4 사진 미션 제출

### Given

팀이 `EXE-50` 미션에 접근했다.

### When

사진 파일을 업로드하고 제출한다.

### Then

- 제출 내역이 생성된다.
- 설정에 따라 즉시 approved 또는 pending_review 처리된다.
- approved 처리 시 50점이 반영된다.
- Exercise 테마가 클리어된다.

## 2.5 스태프 미션 승인

### Given

팀이 `WTR-80` 미션에 접근했고 staff 제출을 생성했다.

### When

관리자가 `/admin/staff`에서 해당 팀과 미션을 성공 처리한다.

### Then

- 제출 상태가 approved로 변경된다.
- 80점이 반영된다.
- Water 테마가 클리어된다.
- audit log가 생성된다.

## 2.6 테마 클리어와 보너스

### Given

팀이 7개 테마를 이미 클리어했다.

### When

마지막 남은 테마의 미션 1개를 성공한다.

### Then

- 8개 테마 모두 클리어 상태가 된다.
- 완주 보너스 100점이 정확히 한 번 지급된다.
- 생명의 열쇠 문구 “홍명기홀로 오라!”가 표시된다.

## 2.7 올클리어 보너스

### Given

팀이 15개 미션을 이미 성공했다.

### When

마지막 16번째 미션을 성공한다.

### Then

- 전체 미션 완료 상태가 된다.
- 올클리어 보너스 200점이 정확히 한 번 지급된다.

## 2.8 최종 인증 실패

### Given

팀이 6개 테마만 클리어했다.

### When

홍명기홀 최종 인증을 시도한다.

### Then

- 인증은 실패한다.
- 추첨권 2장이 추가되지 않는다.
- 남은 테마가 표시된다.

## 2.9 최종 인증 성공

### Given

팀이 8개 테마를 모두 클리어했다.

### When

홍명기홀 최종 QR 인증을 진행한다.

### Then

- 최종 인증이 성공한다.
- final_verified가 true가 된다.
- 추첨권 2장이 추가된다.
- 중복 인증 시 추가 추첨권이 지급되지 않는다.

## 2.10 히든 QR 제한

### Given

팀이 히든 QR을 3개 이미 획득했다.

### When

4번째 히든 QR을 인증한다.

### Then

- 기록은 남길 수 있으나 점수는 추가 지급되지 않는다.
- 화면에 “팀당 최대 3개까지 점수 인정” 안내가 보인다.

## 2.11 숨은 운영진 보너스 제한

### Given

팀이 이미 숨은 운영진 보너스를 받았다.

### When

관리자가 같은 팀에게 숨은 운영진 보너스를 다시 부여하려 한다.

### Then

- 중복 지급이 차단된다.
- 관리자에게 이미 지급된 팀이라는 안내가 표시된다.

## 2.12 추첨권 계산

### Given

팀 점수가 920점이고 최종 인증을 완료했다.

### When

대시보드와 관리자 화면에서 추첨권을 확인한다.

### Then

- 점수 기준 4장 + 최종 인증 2장 = 총 6장으로 표시된다.
- 최대 6장을 초과하지 않는다.

## 3. 관리자 기능 수용 기준

## 3.1 관리자 대시보드

관리자는 `/admin`에서 다음 정보를 확인할 수 있어야 한다.

- 전체 팀 수
- 전체 제출 수
- 승인 대기 제출 수
- 평균 점수
- 완주 팀 수
- 최종 인증 완료 팀 수
- 미션별 제출 현황

## 3.2 제출 검토

관리자는 `/admin/submissions`에서 다음을 할 수 있어야 한다.

- 팀별 제출 조회
- 미션별 제출 조회
- 상태별 필터링
- 제출 승인
- 제출 반려
- 승인 취소
- 메모 입력

## 3.3 스태프 미션 승인

관리자는 `/admin/staff`에서 다음을 할 수 있어야 한다.

- 팀 번호로 검색
- 미션 코드로 필터링
- 성공 처리
- 실패 처리
- 이미 승인된 미션 중복 승인 방지

## 3.4 팀 점수 관리

관리자는 `/admin/teams`에서 다음을 할 수 있어야 한다.

- 팀별 점수 조회
- 테마 클리어 상태 조회
- 최종 인증 여부 조회
- 추첨권 수 조회
- 점수 수동 조정
- 점수 재계산
- CSV export

## 3.5 공지 및 돌발 미션

관리자는 `/admin/announcements`에서 다음을 할 수 있어야 한다.

- 공지 작성
- 공지 활성/비활성
- 돌발 미션 생성
- 돌발 미션 점수 설정
- 돌발 미션 제출 확인

## 4. 테스트 하네스 수용 기준

Codex는 다음 테스트 파일을 만들어야 한다.

```text
tests/e2e/team-flow.spec.ts
tests/e2e/admin-flow.spec.ts
tests/e2e/scoring-flow.spec.ts
tests/load/k6-login.js
tests/load/k6-mission-submit.js
tests/load/k6-final-verification.js
```

테스트는 실제 Supabase 프로젝트가 없어도 최소한 mock 또는 local test mode로 핵심 계산 로직과 UI 흐름을 검증할 수 있어야 한다.

## 5. 빌드 및 품질 기준

다음 명령이 성공해야 한다.

```bash
npm run lint
npm run typecheck
npm run test
npm run test:e2e
npm run build
```

부하 테스트 스크립트는 다음 명령으로 실행 가능해야 한다.

```bash
npm run test:load
```
