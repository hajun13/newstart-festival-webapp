# 부하 테스트 및 안정성 요구사항

## 1. 목표

행사 당일 웹앱 장애는 전체 운영에 직접적인 영향을 준다. 따라서 개발 단계에서 성능, 중복 처리, 장애 대응, 백업 운영을 반드시 고려한다.

본 웹앱은 실제 참가자 약 300명, 팀 약 30개 규모를 기준으로 하되, 테스트는 1.5~2배 수준의 여유를 두고 진행한다.

## 2. 운영 환경 권장

- Supabase Pro 이상 권장
- Vercel Pro 또는 안정적인 배포 환경 권장
- 팀 코드 기반 로그인 사용
- 이메일/문자 인증 사용 금지
- 사진 업로드는 클라이언트 압축 후 업로드
- 영상 업로드는 기본 제외 또는 외부 제출 안내
- 실시간 랭킹은 Supabase Realtime 의존을 피하고 15~30초 polling 권장

## 3. 주요 위험 요소

### 3.1 오프닝 직후 동시 접속

- 참가자가 공통 QR을 동시에 스캔한다.
- 팀장이 동시에 로그인한다.
- 대시보드 조회가 몰린다.

대응:

- 팀 코드 로그인 단순화
- `/dashboard` 쿼리 최소화
- 정적 데이터 캐싱
- 불필요한 realtime 기능 비활성화

### 3.2 사진 업로드 폭주

- 사진 인증 미션에서 이미지 업로드가 몰릴 수 있다.
- 모바일 네트워크 환경이 불안정할 수 있다.

대응:

- 클라이언트 이미지 압축
- 사진 1장 1MB 이하 권장
- 팀장 1명만 업로드하도록 안내
- 업로드 실패 시 재시도 UX 제공
- 최악의 경우 카카오톡/구글폼 백업 제출 허용

### 3.3 미션 제출 중복

- 팀이 제출 버튼을 여러 번 누를 수 있다.
- 네트워크 지연으로 중복 요청이 발생할 수 있다.

대응:

- `unique(team_id, mission_id)` 제약
- 서버에서 idempotent 처리
- 이미 승인된 미션은 다시 점수 부여 금지
- 제출 중 버튼 비활성화

### 3.4 스태프 승인 지연

- 스태프 미션 현장에 팀이 몰릴 수 있다.
- 관리자가 승인 처리를 누락할 수 있다.

대응:

- 관리자 스태프 화면을 단순화
- 팀 번호 검색/필터 제공
- 미션별 승인 대기 목록 제공
- 승인/실패 처리 후 audit log 기록

### 3.5 최종 인증 몰림

- 종료 직전 홍명기홀 QR 인증이 몰릴 수 있다.

대응:

- 최종 인증 API 단순화
- 8개 테마 클리어 여부만 빠르게 검증
- 팀당 1회 unique 처리
- 실패 시 남은 테마 안내

## 4. 성능 목표

## 5. 부하 테스트 시나리오

### 시나리오 A: 오프닝 직후 접속

목표:

- 500명 동시 접속을 가정한다.
- 로그인, 대시보드, 코드 입력 화면 조회를 테스트한다.

경로:

- `/login`
- `/dashboard`
- `/code`

확인 항목:

- 로그인 성공률
- 대시보드 응답 시간
- DB connection 문제
- Vercel function 오류

### 시나리오 B: 미션 제출 폭주

목표:

- 팀들이 동시에 미션 코드를 입력하고 제출하는 상황을 테스트한다.

액션:

- 미션 코드 입력 200회/분
- 퀴즈 제출 200회/분
- 텍스트 제출 100회/분
- 사진 업로드 100건/5분

확인 항목:

- 중복 제출 방지
- 점수 중복 지급 방지
- 업로드 성공률
- DB write 오류

### 시나리오 C: 스태프 승인 집중

목표:

- 스태프 미션이 몰리는 상황을 테스트한다.

액션:

- 승인/실패 처리 50건/분
- 팀별 현황 조회
- 관리자 페이지 필터링

확인 항목:

- 승인 후 점수 반영
- audit log 생성
- 중복 승인 방지

### 시나리오 D: 최종 인증 몰림

목표:

- 행사 종료 직전 홍명기홀 최종 QR 인증이 몰리는 상황을 테스트한다.

액션:

- 최종 인증 100회/분
- 일부 팀은 8개 테마 미완료 상태로 인증 시도
- 일부 팀은 이미 인증 완료 상태로 중복 시도

확인 항목:

- 조건 검증 정확성
- 중복 인증 방지
- 추첨권 계산 정확성

## 7. 구현 안정성 요구사항

1. 모든 쓰기 API는 서버에서 team/session 권한을 검증한다.
2. 모든 점수 부여는 idempotent하게 처리한다.
3. 같은 팀이 같은 미션을 여러 번 성공 처리해도 점수는 한 번만 반영된다.
4. 히든 QR은 팀당 최대 3개까지만 점수를 부여한다.
5. 홍명기홀 최종 인증은 8개 테마를 모두 클리어한 팀만 성공한다.
6. 이미 최종 인증한 팀은 중복 인증이 불가하다.
7. 관리자 점수 수정은 audit log에 기록한다.
8. 제출 취소 또는 반려 시 점수도 정확히 회수되어야 한다.
9. 사진 업로드 실패 시 재시도 안내를 제공한다.
10. 관리자 화면은 자동 새로고침 간격을 최소 10초 이상으로 둔다.

## 8. 장애 대응 플랜

| 장애 상황 | 대응 |
|---|---|
| 사진 업로드 실패 | 카카오톡 또는 구글폼 백업 제출 |
| 특정 미션 코드 오류 | 스태프가 관리자 화면에서 직접 성공 처리 |
| Storage 지연 | 사진 제출 임시 생략 후 스태프 확인 처리 |
| 웹앱 일부 오류 | 관리자 점수 수동 수정 |
| 웹앱 전체 장애 | 구글폼 백업 링크 공개 |
| 네트워크 장애 | 종이 점수표 수기 운영 전환 |
